# OR-Map

Нужно реализовать распределенную Map-структуру OR-Map с политикой Last-Write-Wins на Go с использованием симулятора [Hive](https://github.com/nikitakosatka/hive).

---

## Контекст

В распределенной Map при конкурентных операциях возникает конфликт:

- одна реплика делает `Put(k, v)`;
- другая реплика почти одновременно делает `Delete(k)`.

Если просто обмениваться последними операциями, состояние может стать неконсистентным: значение может возобновиться после удаления, либо удаление может потеряться. В этой лабораторной нужно реализовать `OR-Map`, который сходится у всех узлов за счет state-based репликации и операции `Merge`.

---

## Как реализовать OR-Map

Используйте state-based подход: узлы периодически/по событию обмениваются состояниями (`MapState`) и объединяют их через `Merge`.

### Вариант 1: Логическое время

Использовать логическую версию вида:

- `Counter` - локальный монотонный счетчик изменений;
- `NodeID` - tie-breaker при равных счетчиках.

Правило сравнения версий должно быть детерминированным (например, лексикографический порядок `(Counter, NodeID)`).

### Вариант 2: Физическое время

Можно использовать физические timestamp, но только если обеспечен детерминированный разбор коллизий (например, одинаковый timestamp дополнительно сравнивается по NodeID).

Важно: в `Merge` вы выбираете запись с более новой версией по вашему правилу LWW и никогда не теряете метаданные, нужные для корректного объединения.

---

## Tombstone

Tombstone - это маркер "ключ удален".  
При `Delete(k)` нельзя просто убрать ключ из `MapState`: в этом случае другая реплика со старым `Put` может вернуть удаленное значение обратно.

Нужно реализовать решение с пометками об удалении:
- при удалении записать `StateEntry{Tombstone: true, Version: ...}`;
- в `Merge` tombstone участвует как обычная запись, сравнивается по версии;
- `Get` и `ToMap` не должны показывать ключи, помеченные tombstone.

---

## Реализация

- `Start(ctx context.Context) error`  
  Запускает обработку сообщений узла и фоновую синхронизацию (обмен состоянием между репликами).

- `Put(k, v string)`  
  Записывает новое значение по ключу с новой версией. После изменения состояние должно быть готово к распространению другим репликам.

- `Get(k string) (string, bool)`  
  Возвращает видимое значение. Для отсутствующего ключа и tombstone должен возвращать `ok == false`.

- `Delete(k string)`  
  Помечает ключ tombstone с новой версией.

- `Merge(remote MapState)`  
  Объединяет локальный и удаленный state по LWW-правилу версий.

- `State() MapState`  
  Возвращает копию полного состояния (включая tombstone) для передачи другим узлам.

- `ToMap() map[string]string`  
  Возвращает представление только с живыми значениями (без tombstone).
  
- `Receive(msg *hive.Message) error`  
  Обрабатывает входящие сообщения синхронизации состояния и вызывает `Merge`.

---

## Требования корректности

Реализация должна удовлетворять свойствам:

1. **Сходимость реплик**  
   После обмена состояниями все корректные узлы должны прийти к одинаковому видимому состоянию Map.

2. **Корректность Merge**  
   `Merge` должен быть устойчив к порядку и повторам применения состояния.

3. **Корректность удаления**  
   Удаленный ключ не должен снова появляться из-за прихода старой версии.
